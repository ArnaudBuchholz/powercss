News

2016-04-15: v1.1 released

We have added support for arbitrary-depth conditional CSS and media queries. See the Conditional Expressions section of updated GitHub Documentation. We added 22 regression tests to ensure quality and reliability.

The API remains almost completely backwards compatible in this release which we feel is a major accomplishment given the requirements of conditional expressions. The only exception is that the irregular symbol names containing "bottom" have been made consistent. For example, border_btm has been renamed to border_bottom as the former was needlessly different from the CSS symbol.

What

PowerCSS is a JavaScript library that helps developers create responsive, double-buffered CSS that is optimized for performance. Watch this presentation for an an overview of CSS double-buffering. PowerCSS is all-new code with many additional capabilities and optimizations.

The project is hosted on GitHub and is available as an npm package. It can be installed using npm install powercss.

Why

A significant problem with traditional CSS, whether written by an expert or someone using {less} or Sass, is that it is not written as the application needs it. Styling that is responsive to the user's environment is either limited or simply not possible.

PowerCSS provides tools to write and apply infinitely adjustable CSS based on almost any real-time data available to an application. For example, an application could use PowerCSS to adjust styling based on device orientation, ambient temperature, ambient light, GPS location, heart rate, or time of day. Traditional static CSS can't compete with this flexibility.

We feel that PowerCSS has not only achieved its primary goal, but that it is often better than static CSS in many other respects as well. It provides a simple and familiar API where experienced CSS authors can use their existing skills to be up and running in minutes. When compressed, a PowerCSS solution can download and render faster, and it can speed up some CSS operations by 10x or more. What's not to like?

  • Real-time styling - Create custom styling for every user of your application at any time.
  • Pure JS - Remove the need for any static CSS files.
  • Namespaced - Play well with frameworks, jQuery, other libraries, and third-party JavaScript.
  • Double-buffering - Minimizes page re-flows with this automatic feature. It can speed up some styling changes by more than 10x.
  • Merging and caching - Control when your styling is updated using time-based minimal processing.
  • Mixins - Create custom symbols at multiple levels: virtual stylesheet, virtual cascade, and global. Change a mixin and watch the styles change immediately.
  • Familiar workflow - Leverage your experience with static CSS files using virtual stylesheets and cascades.
  • Machine optimized CSS - Have the browser work more efficiently as only one stylesheet is used for styling at any given time, and numerous redundancies are removed during its preparation.
  • Highly compressible - Compress your styling to a fraction of static CSS.
  • Quality code - Use well tested and documented code. A commit hook is used to ensure no changes occur unless they pass JSLint and regression tests.

Older News

2016-03-25: v1.0

We feel the code is now production ready. It has strong set of regression tests and has been tested in a number of production and production-intent projects with good results. We plan to add per-cascade double-buffered stylesheets in the 1.2.x release.

2016-03-20: v0.7.1

We are very close to 1.x release. The rule-map resolver has been revised to handle arbitrarily complex data structures. This allows us to create mixin maps that support any combination of alternate-lists, concatenations, and symbol-lookups. The isolated code can be found in be found in test/resolve.js. This has been merged with dist/pcss.js, however, we still need to move the tests to test/t.js.

2016-03-09: v0.6.6

Isn't regression testing grand? Just implementing basic coverage has turned up a few interesting bugs and needs. One example: the process update test was sporadically failing about 1/10th of the time because sometimes the test JS proceeded before the CSS string processing completed. So now we have events which are trigged by these processing steps (not yet supported in IE). They are _pcss_merged_, _pcss_prepared_, and _pcss_used_. The event object provides the ID of the cascade affected.

2016-03-07: v0.6.2

Automated regression tests are now implemented. You may run them using npm install and npm test in the root installation directory (node_modules/powercss/). Code coverage metrics and production deployments are underway to help flush out any remaining refinements or bugs we may want to address.

2016-03-04: v0.5.20

Preparations for regression tests. Added key benefits section.

2016-02-24: v0.5.16

This page now showcases double-buffer and theming capabilities. We have also completed the planned 1.x API with the addition of the _getAssetIdList_ methods. At this point, all that remains is integration work and regression tests.

2016-02-24: v0.5.0

PowerCss is nearly feature complete, and we consider it officially Beta. We have been updating examples and squashing niggling bugs as we find them.

This version saw our last big API overhaul we expect before the 1.x release. We cut the number of methods in half while adding many capabilities. The documentation and examples have expanded and updated. We have added the PowerCSS Cookbook section to the GitHub Documentation which we hope you find useful.

The work remaining includes documentation updates, example implementations, and regression tests. We hope to complete those tasks and see a 1.x release before the end of March 2016.
